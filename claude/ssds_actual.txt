1. DOCUMENT PURPOSE
1.1 Purpose 
The Software System Design Specification serves as the comprehensive technical blueprint for the CibiNet food donation platform, establishing the authoritative reference for the system's architecture, functional capabilities, and data management strategies.

The primary objectives of this document are as follows: 
Requirements and Analysis: To translate high-level business requirements into functional requirements and to analyze system behaviors, constraints, and edge cases to ensure the system effectively addresses the coordination gap between food donors and recipients. 
System Design: To define the structural components of the CibiNet platform.
Evolutionary Roadmap: To provide a technical framework that supports the iterative development lifecycle, allowing for incremental delivery while maintaining system integrity. 
This document is intended for stakeholders involved in the lifecycle of the CibiNet project, specifically: 
The development team to guide the implementation of system components. 
System architects to review the integration of third party systems to ensure the scalability of the system 
Quality Assurance Personnel to facilitate the creation of test cases based on the defined functional boundaries and audit requirements. 

2. INTRODUCTION
This document defines the architecture, scope, and objectives of CibiNet, a web-based food donation coordination platform. CibiNet is designed to be a system that facilitates the timely redistribution of surplus food from individual and organizational donors to food banks and shelters. The system emphasizes real-time availability and auditable interactions between participating parties. 

Food donation today is hindered by fragmented communication, manual coordination, and lack of real-time visibility into surplus inventory. These inefficiencies result in wasted food and delayed support for food-insecure populations. CibiNet addresses these needs by providing a centralized, role-based platform where donors can publish surplus food and food banks can immediately discover, claim, and coordinate receipt of those donations. The system directly meets customer needs by reducing coordination latency and improving transparency in food redistribution. 

CibiNet’s primary objectives are to reduce food waste, improve the operational efficiency of charitable food distribution.

From a product perspective, CibiNet is a standalone platform that does not extend or embed into an existing commercial donation system. However, it is designed with clear external interfaces including email, mapping/location, and delivery services needed to operate the application. 
2.1 Scope 
Included Functionality:
Real-time publishing of food donations with quantity, expiration, and fulfillment method (pickup or drop-off)
Automatic expiration and cleanup of unclaimed donations 
Claim confirmation notifications
Audit logging of donation creation and claim events
Search/filtering capabilities allowing users to find available food donations 
Basic compliance-aware features, including expiration tracking and user acknowledgment of food safety responsibility.

Excluded Functionality:
Financial transactions or monetary donations 
Volunteer scheduling or transportation logistics 
Mobile-native applications
Real time logistics: route/driver tracking 
Full regulatory compliance engine

The following diagram defines the CibiNet system boundary and the interfaces crossing that boundary. 

The CibiNet system boundary is defined as the collection of software components under the direct control of the project team. 

The following interfaces cross this boundary:

3. GLOSSARY
Administrator: A privileged system role responsible for managing user accounts, overseeing platform activity, and resolving disputes flagged in the audit log. Administrators are not subject to role-based restrictions applied to standard users.
Audit Log: A permanent record of system actions such as donation creation, claim confirmations, and expiration events.
Claim: An action performed by a food bank or shelter to reserve an available donation for pickup or delivery.
Donor: A user (individual or organization) who posts surplus food donations into the system.
Donation Listing: A published entry containing food details such as quantity, type, expiration date, and pickup/drop-off method.
Expiration: The automatic removal or deactivation of a donation listing after its expiration time is reached or if it remains unclaimed.
Food Bank / Shelter: A user entity responsible for receiving and distributing donated food.
Fulfillment Method: The method by which a donation is transferred (pickup by recipient or drop-off by donor).
Notification: A system-generated message (email or in-app) sent to users to confirm events such as donation posting, claiming, or expiration.
Role-Based Access Control (RBAC): A security model restricting user actions based on their assigned role (donor vs recipient).
Surplus Food: Food that is safe for consumption but is excess and would otherwise be discarded.

5. BUSINESS REQUIREMENTS 
CibiNet is designed to address the inefficiency and lack of coordination in food donation systems. The system must provide a centralized platform that enables donors to post available food donations and recipients to quickly discover and claim those donations. Business requirements prioritize transparency, rapid coordination, and reduced food waste.
The system must be accessible through a web browser and support multiple user roles. It must also maintain traceability of donation and claim activity through audit logs to ensure accountability and trust between participants.
5.1 Technology 
CibiNet must be implemented as a modern web-based platform that supports real-time donation discovery and role-based user interaction. The system must include a backend service for business logic, a database for persistent storage, and a frontend interface for user interaction.
The system must support integration with external services including:
Email services for notifications
Mapping/geolocation services for pickup/drop-off coordination
The system must be deployable on commodity cloud infrastructure and must support future scalability.
5.2 Economics 
CibiNet aims to mitigate the economic impact of food waste by improving the efficiency of food redistribution. The system provides value by allowing donors to reduce disposal costs while improving access to food for recipients.
The system is designed to be low-cost to operate, using cloud-based deployment and open-source tools where possible. The expected cost drivers include hosting fees, database storage, and third-party API usage (email and mapping services). The system must minimize operational cost while maintaining acceptable performance and availability.
5.3 Regulatory and Legal 
CibiNet does not directly handle food preparation, storage, or physical transfer. However, the platform may be used in contexts where food safety regulations apply. Therefore, the system must include basic data validation and disclaimers indicating that donors and recipients are responsible for ensuring food safety compliance.
CibiNet must also follow general data privacy expectations. The system must protect user information such as contact details and location data. User authentication and access control must be enforced to prevent unauthorized access to sensitive information.
5.4 Market Considerations 
CibiNet is designed for individuals, restaurants, grocery stores, and organizations that frequently generate surplus food, as well as food banks and shelters that distribute food to communities in need.
Existing food donation systems are often fragmented and rely on manual communication methods such as phone calls, spreadsheets, and social media posts. CibiNet provides competitive advantage by offering centralized visibility, real-time claiming, and transparent tracking through audit logs.
The platform must be simple to use, require minimal training, and provide clear incentives for both donors and recipients to participate.

5.6 Human Resources and Training 
CibiNet requires a development team capable of implementing a full-stack web application, including frontend development, backend services, database design, and deployment.
Users of the system (donors and food banks) are expected to have basic web navigation skills. Minimal training should be required due to the system’s intuitive interface design.
Administrative training is not required, since CibiNet is designed to operate as a self-service platform. However, documentation should be provided for users to understand donation posting, claiming, and notification workflows.
6. USER REQUIREMENTS (DESCRIPTIVE - FUNCTIONAL, AND NON-FUNCTIONAL REQUIREMENTS)
This section contains all software requirements for the CibiNet platform in sufficient detail to enable designers to design a system that satisfies them and testers to test the system to meet them. Users, operators, or external systems perceive every requirement externally. 
6.1 Functional Descriptive Detailed Requirements 
FR-01: User Authentication 
Definition: The capability for the system to verify user identities and enforce role-based access.
Specifications: 
The system shall provide a secure login interface for registered users.
The system shall validate credentials against the central database using SHA-256 hashing. 
The system shall grant access to specific features based on the user’s assigned role (Donor/Recipient).
FR-02: Donation Cataloging
Definition: The capability for Donors to digitalize and list surplus food inventory into the centralized platform 
Specifications: 
The system shall provide and input form for Donors to enter food category, description, and quantity 
The system shall require the Donor to input a mandatory safety expiration timestamp (“Best Before” date) for each listing. 
The system shall trace thai entry to the central database to make it immediately available for discovery 
FR-03: Geographic Discovery
Definition: The capability for Recipients to find available donations based on geographic proximity 
Specifications: 
The system shall render a map-based interface displaying active food donations. 
The system shall filter and sort available donations based on the Recipient’s entered address. 
The system shall allow filtering by food type (e.g. perishable vs non-perishable))
FR-04: External Logistics Handoff 
Definition: The capability to export pickup and drop-off coordination data to third-party delivery/transportation services. 
Specifications: 
The system shall generate a digital Logistics Packet containing validated pickup and drop-off coordinates once a delivery is required. 
The system shall provide a data interface to pass location data directly to external delivery applications. 
FR-05: Donation Claiming 
Definition: The capability for a Recipient to exclusively reserve a donation item, preventing duplicate claims. 
Specifications: 
The system shall enforce an atomic “Claim-Lock” on available inventory upon Recipient action. 
The system shall immediately update the listing status to “Claimed” and remove it from the public discovery map. 
The system shall display an error message if a user attempts to claim an item that has just been locked by another user. 
FR-06: Automated Safety Pruning
Definition: The automated removal of unsafe or expired good listings from the active catalog. 
Specifications: 
The system shall execute a scheduled background process that continually monitors expiration timestamps of all active listing s
The system shall automatically change the status of any listing that surpassed its safety window to “Expired.”
The system shall permanently hide “Expired” items from the discovery interface. 
FR-07: Fulfillment Notification Dispatch 
Definition: The automated communication of transaction and coordination details to involved stakeholders 
Specifications: 
The system shall trigger a dual-party email notification immediately upon a successful claim lock. 
The system shall include a unique Coordination ID, involved party contact information, and chosen fulfillment instructions in the notification.
6.2 Non-Functional Descriptive Detailed Requirements
NFR-01: Performance 
Definition: Constraints regarding the responsiveness and speed of the system 
Specifications: Search queries and map rendering for local donations must be processed and returned to the user interface within 3 seconds under normal load assuming a reliable network connection. 
NFR-02: Reliability and Availability
Definition: Constraints regarding uptime and fault tolerance
Specifications: The web platform shall maintain a 99.5% uptime during core operation horus to ensure continuous coordination for perishable goods. 
NFR-03: Security and Privacy 
Definition: Constraints regarding the protection of user data and system access. 
Specifications: 
The system shall enforce Role-Based Access Control (RBAC), restricting donation creation to Donors and claiming actions to Recipients. 
Donor street addresses and direct contact information must be obfuscated from the public map until a Recipient successfully claims the item.
NFR-04: Usability 
Definition: Constraints regarding the user experience and interface accessibility 
Specifications: The web interface must be fully responsive, functioning identically on standard desktop browsers without requiring specialized administrative training 
7. SYSTEM ARCHITECTURE
This section presents a high-level overview of the CibiNet system architecture, illustrating how functions are distributed across system modules to facilitate the secure and rapid redistribution of surplus food. 

CibiNet follows a Client-Server architecture. The system operates centrally, with distinct logical layers separating the user interface, business logic, and data persistence. 

System Modules: 
Frontend - Presentation Module: A responsive, web-based user interface accessed via standard browsers. It is responsible for rendering the map-based discovery interface and donation input forms. The frontend communicates with the backend exclusively through secure API endpoints.
Backend - Application Logic Module: The core service engine handling all business rules. It contains sub-modules for:
Concurrency Management: Handling the atomic “Claim-Lock” logic to prevent double-booking.
Background Scheduler: Running the automated safety pruning for expired food.
Notification Engine: Assembling transaction details for dispatch.
Audit and Transaction Logging Module: Maintains immutable records of user actions and claim transactions to support transparency and traceability.
Data Storage Module: A relational database responsible for persistently storing user account information, active food inventory, and read-only audit logs. The database enforces referential integrity and supports real-time querying of donation availability.
Reused Architectural Components - External Integration Module: Although CibiNet is a standalone platform and does not embed into existing commercial donation systems, it integrates with established third-party services to support operational needs without duplicating complex infrastructure. These include:
Mapping/Geocoding API: Used to validate donor addresses and render proximity maps.
Third-Party Delivery API: Accepts structured logistics data and coordinates physical transportation of food donations.
Email Gateway Service: Dispatches dual-party fulfillment notifications and transaction confirmations.

Donation Claiming

Automated Safety Pruning
Description

Frontend Component (React/Vite :3000): Contains the presentation layer sub-components — AuthContext, MapView, ListingForm, ClaimUI, NotificationUI, and the api/client.ts HTTP client. The frontend has no direct database access; all data flows through api/client.ts.

Backend Component (TurboGears WSGI :8080): Contains the application logic sub-components — RootController, ApiController, AuthController, ListingController, ClaimController, NotificationController, SystemController, UserController, and the cleanup_scheduler() daemon thread. URL routing is purely object-based (no decorator route tables); TurboGears maps URL path segments to controller class attributes.

Data Layer (SQLAlchemy + SQLite): Contains five ORM model classes (User, Listing, Claim, Notification, AuditLog) and schemas.py, which is the sole response serialization layer. All controllers import from model.py and schemas.py; no controller constructs response dicts inline.

External Services: Nominatim Geocoding API (address → lat/lon; no API key required). Email Gateway (SMTP) is architecturally supported but not yet implemented.

10.2 Software Architecture General Description
CibiNet follows a three-tier client-server architecture:

Presentation Tier: A React 19 single-page application built with TypeScript and Tailwind CSS, served via Vite. Authentication state is managed globally through AuthContext.tsx using React context. Route protection is enforced by ProtectedRoute.tsx, which reads from useAuth().

Application Logic Tier: A TurboGears 2 Python application running as a WSGI process on port 8080. The controller hierarchy (RootController → ApiController → sub-controllers) maps directly to the URL hierarchy (/api/auth/*, /api/listings/*, etc.). A daemon thread (cleanup_scheduler) runs inside the same process to execute safety pruning every 300 seconds.

Data Tier: SQLAlchemy ORM over a SQLite file. A single shared session object (imported from model.py) serves all web request handlers. An isolated per-call session is used by run_cleanup() because SQLAlchemy sessions are not thread-safe.

The rationale for this decomposition: the controller hierarchy directly mirrors the functional component breakdown, making it possible to add or modify a feature in a bounded set of files without changing routing configuration. See Section 14 for detailed rationale on each key decision.

10.3 Software Item Components
The following components are identified and their primary functionality described:

AuthController (controllers/auth.py): Handles user registration, login, logout, and session status checks. Enforces SHA-256 password hashing. Reads and writes tg_session to persist role-based identity across requests.

ListingController (controllers/listings.py): Handles all donation lifecycle operations — create, nearby discovery (bounding box query), claim (atomic), update, delete, and detail. Geocodes addresses via Nominatim. Enforces RBAC at each endpoint. Orchestrates the claim transaction including notification dispatch and audit logging.

ClaimController (controllers/claims.py): Provides Recipients with read access to their own claim records via ClaimSchema, which exposes the full donor address and Coordination ID post-claim.

NotificationController (controllers/notifications.py): Provides in-app notification retrieval and read-state management. Also contains dispatch_claim_notifications(), the module-level function called by ListingController during the claim transaction.

SystemController (controllers/system.py): Exposes manual triggers for safety pruning and audit log retrieval. Contains run_cleanup(), which is also called by the daemon thread.

UserController (controllers/users.py): Provides users access to their own profile and listing history.

Background Scheduler (main.py — cleanup_scheduler()): A daemon thread started at application startup. Wakes every 300 seconds and calls run_cleanup() using an isolated SQLAlchemy session.

schemas.py: The sole response serialization layer. Contains UserSchema, ListingMapSchema, ListingSchema, LogisticsPacketSchema, ClaimSchema, NotificationSchema, and AuditLogSchema. Controllers never construct response dicts inline.

model.py: Defines all ORM model classes and the SQLAlchemy engine/session bindings. The shared session instance is imported into every controller.

10.5 Software Component Concept Of Execution
AuthController: Enters execution on every HTTP request to /api/auth/*. Triggered by the user submitting login or registration credentials. Expected outcome: session is established (login) or cleared (logout), and the appropriate JSON response is returned. No background execution.

ListingController: Enters execution on every HTTP request to /api/listings/*. The claim() method is the most critical: it enters a database transaction, acquires a logical lock by filtering on status='available', and exits by committing all changes atomically. Expected outcome: listing status is updated, claim record is persisted, notifications are staged, and audit log is written — all in one commit.

NotificationController (dispatch_claim_notifications): Enters execution synchronously within the ListingController claim transaction. Triggered when a claim is successfully flushed and claim_id is available. Expected outcome: two Notification records staged in the shared session. The function does not commit; the calling controller commits.

SystemController (run_cleanup): Enters execution either (a) via the daemon thread every 300 seconds, or (b) on a manual GET /api/system/cleanup request. Uses an isolated SQLAlchemy session to avoid thread-safety issues. Expected outcome: all available listings with expiry_time in the past are marked expired, and one AuditLog record is written per expired listing.

cleanup_scheduler (daemon thread): Enters execution at application startup (main.py). Runs an infinite loop with time.sleep(300). Expected outcome: automatic background maintenance of listing freshness with no user intervention. Terminates automatically when the WSGI process exits (daemon=True).

listing_created — triggered by ListingController.create() on successful commit.
listing_claimed — triggered by ListingController.claim() on successful commit.
listing_expired — triggered by run_cleanup() for each listing whose expiry_time has elapsed.
listing_updated — triggered by ListingController.update() on successful commit.
listing_deleted — triggered by ListingController.delete() on successful commit.

11.2.1 Run-time Object Instances
At runtime, the following object instantiation relationships exist:

The shared SQLAlchemy session (model.session) is instantiated once at module load time and shared across all web-request handler invocations within a single process. This is the standard TurboGears session-per-application pattern.

The cleanup_scheduler() daemon thread instantiates one new isolated SQLAlchemy Session() per invocation of run_cleanup(). This session is created, used, committed, and closed within the scope of a single cleanup cycle. This isolation prevents the scheduler from sharing state with the web-request session.

The TurboGears controller objects (AuthController, ListingController, etc.) are instantiated once at application startup as class attributes of ApiController. They are stateless — all per-request state flows through tg_session and the SQLAlchemy session.

Schema objects (e.g., ListingMapSchema, NotificationSchema) are instantiated per-response by the controller, wrap a model object, and are discarded after to_dict() is called. They carry no persistent state.

11.4 Concept of Execution
ListingController.claim(): The claim endpoint enters execution when a Recipient's browser sends GET /api/listings/claim. It performs sequential guard checks (authentication → role → logistics_type validity → listing availability), then opens an implicit SQLAlchemy transaction. The transaction mutates listing.status, creates the Claim record, stages two Notification records, and writes an AuditLog entry. The commit() call is the point at which all mutations become durable and visible to other sessions. If any step raises an exception, rollback() returns the database to its pre-claim state and a 500 error is returned.

cleanup_scheduler(): Enters execution when main.py calls threading.Thread(target=cleanup_scheduler, daemon=True).start() at application startup. It sleeps for 300 seconds, then delegates to run_cleanup() which opens and closes its own isolated session. The scheduler itself does not hold any session state between cycles.

11.5.1 Unique Identifier of Interface

11.5.2 Interface Identification and Diagrams
INF-01 — HTTP API Interface: Implemented as HTTP GET/POST requests with JSON response bodies. The frontend api/client.ts sends requests to relative paths (e.g., /api/listings/nearby) which are proxied in development and forwarded by Nginx in production. This is a standard HTTP interface requiring no additional specification beyond the endpoint table in Section 16.3.

INF-03 — ORM Data Interface: Implemented via the SQLAlchemy session API. Controllers call session.query(), session.add(), session.flush(), session.commit(), and session.rollback(). INF-05 — Geocoding API Interface: Implemented in listings.py (geocode_address()). Constructs a URL-encoded GET request to https://nominatim.openstreetmap.org/search with parameters q (address string), format=json, and limit=1. Returns the first result's lat and lon as floats. On any exception (timeout, no results, network error), falls back to hardcoded defaults (40.7128, -74.0060). This is a reusable interface component callable from any controller requiring geocoding.

INF-06 — Notification Dispatch Interface: The dispatch_claim_notifications(claim, listing, donor, recipient) function signature defines the interface. It accepts four SQLAlchemy model objects and adds two Notification records to the shared session without committing. The caller is responsible for the transaction boundary. If an SMTP email gateway is added in the future, the addition is isolated to this function without changes to ListingController.

14. RATIONALE
This section documents the reasoning behind key design decisions made during implementation of CibiNet.

Object-Based URL Routing (TurboGears 2)
TurboGears maps URL segments to Python class attributes rather than using decorator-based route tables. This was chosen because it enforces a clean, hierarchical controller structure that mirrors the system's logical component breakdown (ApiController → AuthController, ListingController, etc.) without requiring a separate routing configuration file.

Cookie-Based Session Authentication (No JWT)
Authentication state is stored in server-side Beaker sessions identified by a signed cookie, not a Bearer token or JWT. This decision eliminates the need for a token revocation mechanism and keeps the implementation self-contained within TurboGears' built-in session middleware. The trade-off is that horizontal scaling requires sticky sessions or a shared session store, which is acceptable given the single-server deployment model.

SQLite for the Database
SQLite was chosen for its zero-configuration setup and suitability for a single-server academic deployment. The entire database is a single file (cibinet_dev.db), which simplifies seeding, resetting, and testing. The SQLAlchemy ORM abstracts the database dialect, so migration to PostgreSQL requires only a DB_URL change.

session.flush() Before Notification Dispatch
Within the claim transaction in controllers/listings.py, session.flush() is called after adding the Claim object but before calling dispatch_claim_notifications(). This is required because flush() assigns the auto-incremented claim_id from the database without committing the transaction, allowing the coordination ID (COORD-{claim_id}) to be embedded in notification messages while keeping the entire claim-notification-audit-log operation atomic.

Address Obfuscation in ListingMapSchema
The schemas.py ListingMapSchema class deliberately excludes the address_text field. Only latitude and longitude are exposed for map rendering. The full address is revealed only via ClaimSchema after a successful claim. This directly implements NFR-03, protecting donor contact and location information from unauthenticated or unclaimed access.

Background Daemon Thread for Safety Pruning
The cleanup_scheduler() function in main.py runs as a daemon thread that wakes every 300 seconds to call run_cleanup(). Using a daemon thread means the scheduler is automatically terminated when the main process exits, requiring no explicit cleanup code. The scheduler uses its own isolated SQLAlchemy session (Session()) rather than the shared web-request session because SQLAlchemy sessions are not thread-safe.

schemas.py as the Sole Response Serializer
All API responses are constructed by schema classes in schemas.py rather than by inline dict construction in controllers. This ensures that field names, field inclusion/exclusion decisions (such as the address obfuscation rule), and response shapes are defined in a single authoritative location and are not duplicated or inconsistently applied across controllers.

15. NOTES
The geocoding step in the listing creation flow (controllers/listings.py, create endpoint) is currently stubbed with a hardcoded coordinate (lat=40.7128, lon=-74.0060, representing New York City). A production deployment must replace this stub with a real geocoding API call (e.g., Google Maps Geocoding API or Mapbox) to convert the donor's address_text into accurate latitude/longitude values.

The MOCK_AUTH flag in frontend/src/context/AuthContext.tsx is currently set to true, which bypasses the real backend and stores a fake user in localStorage. To connect the frontend to the real backend authentication flow, MOCK_AUTH must be set to false and the TODO blocks in that file must be implemented.

Email notification dispatch is implemented as in-app Notification database records only. The UC07 requirement for dual-party email is architecturally supported (the notification message body contains the Coordination ID, contact information, and fulfillment method) but the SMTP gateway integration has not been implemented. An SMTP client would be added to dispatch_claim_notifications() in controllers/notifications.py.

The session validation key ('cibinet-dev-secret-key') in main.py is a development placeholder and must be replaced with a cryptographically random secret before any production deployment.

16.1 System Test Plan Requirements
This section describes the SQA testing process, product testing, and acceptance testing for CibiNet.

Testing is conducted against the running full-stack system (backend on port 8080, frontend on port 3000) with a freshly seeded database (python db_seed.py). No mocks are used for database interactions — all tests operate against the real SQLite instance.

TC01 — User Authentication (FR-01, UC01)
Scenario: Valid Donor login.
Steps: POST credentials for a seeded Donor account to /api/auth/login. Verify response status is "success" and the returned user object contains role "Donor". Verify session cookie is set.
Pass Criteria: HTTP 200, status == "success", role == "Donor".

TC02 — Invalid Credentials Rejection (FR-01, UC01 Alternative Flow)
Scenario: Login with wrong password.
Steps: POST /api/auth/login with a valid email and incorrect password.
Pass Criteria: HTTP 401, response contains "error" key.

TC03 — Create Donation Listing (FR-02, UC02)
Scenario: Donor creates a listing.
Steps: Authenticate as Donor. POST to /api/listings/create with food_type, quantity, address_text, hours_until_expiry. Verify listing appears in the donor's listing view.
Pass Criteria: HTTP 200, listing_id returned, listing visible via /api/users/my_listings.

TC04 — Geographic Discovery (FR-03, UC03)
Scenario: Recipient discovers nearby listings.
Steps: Authenticate as Recipient. GET /api/listings/nearby with lat, lon, radius=5. Verify response contains listings array. Verify address_text is NOT present in any listing object (NFR-03).
Pass Criteria: HTTP 200, listings array non-empty, no address_text field in any listing.

TC05 — Atomic Donation Claiming (FR-05, UC05)
Scenario: Recipient claims an available listing.
Steps: Authenticate as Recipient. GET /api/listings/claim with a valid listing_id and logistics_type=self_pickup. Verify status changes to "claimed". Verify Coordination ID is returned. Verify listing no longer appears in /api/listings/nearby.
Pass Criteria: HTTP 200, coordination_id present, listing absent from discovery results.

TC06 — Double-Claim Prevention (FR-05, UC05 Alternative Flow)
Scenario: Two simultaneous claim attempts on the same listing.
Steps: Claim a listing successfully (TC05). Attempt a second claim on the same listing_id.
Pass Criteria: Second request returns HTTP 409 with "Item unavailable" error.

TC07 — Automated Safety Pruning (FR-06, UC06)
Scenario: Expired listing is removed from discovery.
Steps: Create a listing with hours_until_expiry=0 (or directly insert a past expiry_time into the database). Trigger /api/system/cleanup. Verify the listing status is "expired" in the database. Verify listing does not appear in /api/listings/nearby.
Pass Criteria: expired_items_removed >= 1, listing absent from discovery.

TC08 — Notification Dispatch (FR-07, UC07)
Scenario: Both parties receive notifications after a claim.
Steps: Claim a listing (TC05). Authenticate as Donor and GET /api/notifications/mine. Authenticate as Recipient and GET /api/notifications/mine.
Pass Criteria: Donor notification of type "claim_received" exists. Recipient notification of type "claim_confirmed" exists. Both contain the matching Coordination ID.

TC09 — Role Enforcement (NFR-03)
Scenario: Donor cannot claim; Recipient cannot create.
Steps: Authenticate as Donor. Attempt GET /api/listings/claim. Authenticate as Recipient. Attempt GET /api/listings/create.
Pass Criteria: Both return HTTP 403 with appropriate "error" message.

TC10 — Audit Log Integrity
Scenario: Actions are recorded in the audit log.
Steps: Perform TC03 (listing creation) and TC05 (claiming). GET /api/system/audit_log.
Pass Criteria: Audit log contains entries with action=="listing_created" and action=="listing_claimed" referencing the correct entity_ids.

16.6 Dictionaries

Data Dictionary — CibiNet System Entities

Entity: User
Field: user_id | Type: Integer (PK) | Description: Auto-incremented primary key uniquely identifying each registered user.
Field: email | Type: String(255), UNIQUE, NOT NULL | Description: The user's email address; serves as the login credential and contact identifier.
Field: password_hash | Type: Text, NOT NULL | Description: SHA-256 hex digest of the user's plaintext password. Never stored or transmitted in plain form.
Field: role | Type: String(20), NOT NULL | Description: The user's role within the system. Allowed values: 'Donor', 'Recipient'. Enforces RBAC across all controllers.
Field: created_at | Type: DateTime | Description: UTC timestamp of account creation. Defaults to datetime.utcnow().

Entity: Listing
Field: listing_id | Type: Integer (PK) | Description: Auto-incremented primary key uniquely identifying each donation listing.
Field: donor_id | Type: Integer (FK → users.user_id), NOT NULL | Description: References the Donor who created the listing.
Field: food_type | Type: String(100), NOT NULL | Description: Category or description of the donated food (e.g., "Canned Goods", "Prepared Meals").
Field: quantity | Type: String(100), NOT NULL | Description: Human-readable description of quantity (e.g., "20 cans", "3 trays").
Field: status | Type: String(20) | Description: Current lifecycle state of the listing. Allowed values: 'available', 'claimed', 'expired'. Defaults to 'available'.
Field: expiry_time | Type: DateTime, NOT NULL | Description: UTC timestamp after which the listing is automatically pruned by the safety scheduler.
Field: latitude | Type: Numeric(9,6) | Description: Geographic latitude of the donation pickup location. Currently stubbed to 40.7128 (NYC); to be populated by geocoding API.
Field: longitude | Type: Numeric(9,6) | Description: Geographic longitude of the donation pickup location. Currently stubbed to -74.0060 (NYC).
Field: address_text | Type: Text, NOT NULL | Description: Full human-readable pickup address. Exposed only via ClaimSchema post-claim; excluded from ListingMapSchema.

Entity: Claim
Field: claim_id | Type: Integer (PK) | Description: Auto-incremented primary key; forms the basis of the Coordination ID (COORD-{claim_id}).
Field: listing_id | Type: Integer (FK → listings.listing_id), UNIQUE, NOT NULL | Description: References the claimed listing. UNIQUE constraint enforces the atomic one-claim-per-listing rule.
Field: recipient_id | Type: Integer (FK → users.user_id), NOT NULL | Description: References the Recipient who claimed the listing.
Field: logistics_type | Type: String(50), NOT NULL | Description: Fulfillment method selected by the Recipient. Allowed values: 'self_pickup', 'third_party'.
Field: external_ref_id | Type: String(100), NULL | Description: Reference ID from a third-party delivery service, if applicable. Null for self-pickup claims.
Field: safety_ack_received | Type: Boolean | Description: True if the Recipient has acknowledged food safety responsibility via the /claims/acknowledge endpoint. Defaults to False.
Field: claimed_at | Type: DateTime | Description: UTC timestamp of the claim action. Defaults to datetime.utcnow().

Entity: Notification
Field: notification_id | Type: Integer (PK) | Description: Auto-incremented primary key.
Field: user_id | Type: Integer (FK → users.user_id), NOT NULL | Description: The user to whom the notification is addressed.
Field: type | Type: String(50), NOT NULL | Description: Notification event type. Values in use: 'claim_received' (sent to Donor), 'claim_confirmed' (sent to Recipient).
Field: message_body | Type: Text, NOT NULL | Description: Human-readable notification text including the Coordination ID, contact info, and fulfillment instructions.
Field: sent_at | Type: DateTime | Description: UTC timestamp of notification creation. Defaults to datetime.utcnow().

Entity: AuditLog
Field: log_id | Type: Integer (PK) | Description: Auto-incremented primary key.
Field: user_id | Type: Integer (FK → users.user_id), NULL | Description: The user who triggered the event. NULL for system-initiated events (e.g., scheduler-driven expiration).
Field: action | Type: String(100), NOT NULL | Description: The recorded action. Values in use: 'listing_created', 'listing_claimed', 'listing_expired'.
Field: entity_type | Type: String(50), NOT NULL | Description: The type of entity affected. Values: 'listing', 'claim'.
Field: entity_id | Type: Integer, NOT NULL | Description: The primary key of the affected entity (listing_id or claim_id).
Field: timestamp | Type: DateTime | Description: UTC timestamp of the logged event. Defaults to datetime.utcnow().
